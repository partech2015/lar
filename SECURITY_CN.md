# 安全与合规性

Lár 被设计为“黑盒”代理框架的“玻璃盒子”替代方案。
对于受监管行业（金融科技、医疗保健、国防）的企业部署，我们优先考虑**确定性**、**可审计性**和**物理隔离（气隙）能力**。

---

## 1. 安全设计

### A. 不可篡改的审计日志（合规性）
每个 Lár 代理在每次运行时都会生成一个可进行密码学链接的 JSON 飞行日志 (`lar_logs/*.json`)。
*   **记录内容**：每个输入、提示词、工具执行和输出。
*   **原因**：这使得满足 21 CFR Part 11 合规性成为可能。您可以准确证明代理 *为什么* 采取了某项行动。
*   **机制**：状态转换是不可变的。

### B. 物理隔离与数据主权
Lár 与互联网完全解耦。
*   **无遥测**：引擎不向 Snath AI 或任何第三方发送任何数据。
*   **本地模型**：只需一行代码即可从 `gpt-4` 切换到 `ollama/phi4` 或 `deepseek-coder`。
*   **离线模式**：无需向云提供商发送任何字节即可验证代理的安全性。

### C. 模式 10：安全防火墙
我们提供了一种标准的架构模式（模式 10）来防止越狱。
*   **机制**：`GuardrailNode` (正则/关键字) -> `Router` -> `LLM`。
*   **益处**：在恶意输入（例如，“忽略之前的指令”）到达 LLM *之前* 将其阻断，从而节省成本并防止未经授权的操作。
*   **示例**：参见 [`examples/10_security_firewall.py`](examples/10_security_firewall.py)。

## 2. 支持的版本

安全补丁和错误修复适用于 Lár 引擎框架的 `最新次要版本`。

| 状态      | 版本格式                               | 安全支持                                         |
|-------------|-----------------------------------------------|-----------------------------------------------------------|
| 支持中     | 最新次要版本（例如 1.0.x）            | 提供完整的补丁和关键错误修复。         |
| 不再支持     | 之前的次要版本（例如 0.x.x）         | 不再发布新的安全补丁。用户应立即升级。 |

## 3. 报告漏洞

我们非常重视 *Lár 引擎* 框架及其用户的安全。如果您发现漏洞，我们要求您遵循 **负责任的披露** 流程。

- **报告渠道**：请仅通过电子邮件向以下人员报告漏洞：[**axdithya@snath.ai**](mailto:axdithya@snath.ai) 或 [**vinay@snath.ai**](mailto:vinay@snath.ai)
- **响应时间**：我们承诺在 24 小时内确认收到您的邮件，并在 48 小时内对漏洞进行初步评估。
- **公共披露**：**请勿打开公共 GitHub Issue、Pull Request，或在公共论坛（如社交媒体或 Discord）上讨论漏洞**。在修复程序发布之前进行公开披露会危及所有用户。
