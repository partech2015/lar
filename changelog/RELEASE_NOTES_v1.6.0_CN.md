# Lár v1.6.0：内存压缩与经济约束

我们很高兴地宣布 **Lár v1.6.0** 正式发布，它为任务关键型智能体引入了强大的全新防御机制任务。

## 问题背景
随着智能体变得更加自治（如 v1.5 引入的 `DynamicNode`），它们面临两个风险：
1. **上下文膨胀 (Context Bloat)**：并行工作节点返回的海量文档淹没了下游的上下文窗口（即“黑洞”现象）。
2. **死循环 (Infinite Loops)**：智能体陷入自我纠错循环并耗尽 API 额度。

## 解决方案

### 1. 内存压缩 (`ReduceNode`)
新的 `ReduceNode` 原生支持 Map-Reduce 模式，以防止上下文窗口膨胀。
它从 `GraphState` 中读取多个繁重的文档，生成一份简明的执行摘要，然后**显式地从内存中删除原始数据键**。
这可以确保状态“接力棒”保持轻量，防止下游模型因巨大的上下文规模而出现卡顿或幻觉。

### 2. 经济约束（Token 预算）
您现在可以为智能体的执行成本设定一个严格的、数学上的上限。
只需在 `initial_state` 中添加一个 `token_budget` 整数即可。每次 `LLMNode` 运行时，它都会从该预算中扣除使用量。如果预算归零，该节点将拒绝执行。现在，您可以从数学上保证一个智能体永远不会超过特定的金额，无论其动态图变得多么复杂。

### 3. 节点疲劳（死循环保护）
`GraphExecutor` 现在可以精确追踪在单词执行轨迹中某个节点被访问的次数。如果智能体在同一个节点上循环（例如“修复代码”节点）超过 `max_node_fatigue` 限制（默认：20 次），执行器将触发熔断器并安全终止运行，同时记录疲劳错误。
