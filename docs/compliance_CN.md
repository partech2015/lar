# 合规性与安全（适配欧盟 AI 法案 - 2026年8月）

> **法律声明**：Lár 框架提供旨在*辅助*合规的架构模式。它本身不保证系统合规。您应对系统的最终验证负责。

Lár 专为满足 **2026 年欧盟 AI 法案 (EU AI Act)** 和 **FDA 21 CFR Part 11** 对高风险 AI 系统的严苛要求而设计。

与那些由于黑盒化导致决策路径模糊的框架不同，Lár 是一个专为取证审计而设计的“玻盒”（Glass Box）引擎。

---

## 欧盟 AI 法案对齐

欧盟 AI 法案（2026 年 8 月全面实施）对“高风险”系统（如医疗器械、雇佣、信用评分、关键基础设施）施加了严格义务。

### 1. 第 12 条：记录保存（日志记录）
**要求**：系统必须能够“在生命周期内自动记录事件（‘日志’）”，以确保可追溯性。

**Lár 解决方案**：`State-Diff Ledger`（状态差异账本）

每一个 Lár 智能体都会自动生成一份 `flight_recorder.json` 日志。这不仅仅是一个简单的调试打印流，而是一个包含以下内容的**取证账本**：

*   **时间戳**：对齐 UTC 的执行时间。
*   **输入/输出**：发送的确切提示词和接收到的原始补全结果。
*   **模型 ID**：所使用的特定模型版本（例如 `gpt-4-0613`）。
*   **状态差异**：内存中具体变量的更改。

```json
// Lár 账本条目示例
{
  "step": 4,
  "node": "TriageNode",
  "timestamp": "2026-08-12T10:00:01Z",
  "state_diff": {
    "risk_score": {"old": 0.1, "new": 0.95},
    "routing": {"old": null, "new": "HUMAN_REVIEW"}
  }
}
```

### 2. 第 13 条：透明度与解释性
**要求**：高风险 AI 系统的设计必须“确保其运行具有足够的透明度，以便用户能够解释系统的输出”。

**Lár 解决方案**：“玻盒”架构

*   **无隐藏提示词**：Lár 不会在背后偷偷注入“系统提示词”。您拥有 100% 的提示词掌控权。
*   **显式路由**：逻辑流定义在标准的 Python 代码（节点与边）中，而不是隐藏在神经网络或复杂的“智能体执行器”循环里。
*   **解释性**：任何 Python 开发者都能读懂 `graph.add_edge("Triage", "Human")` 并理解决策路径，而无需理解 LLM 的内部权重。

### 3. 第 14 条：人类监督
**要求**：系统设计必须确保“自然人能够进行有效监督”，包括“中断系统”或“撤销”决策的能力。

**Lár 解决方案**：原生 `Interrupt`（中断）模式

Lár 将“人工干预”视为图中的一等公民。

*   **暂停与恢复**：您可以在检查点（例如 `before="ExecuteTool"`）之前执行图，检查状态并恢复执行。
*   **状态修改**：人工监督员可以在批准下一步之前手动编辑内存（例如修正幻觉生成的参数）。

## FDA 21 CFR Part 11 (电子记录)

对于医疗保健和制药应用（例如何药物研发流水线），Lár 支持 **Part 11** 的关键要求：

1.  **验证**：图的确定性本质意味着您可以运行回归测试。给定输入 X 和固定种子 Y，图走过的路径基本一致。
2.  **审计轨迹**：`GraphExecutor` 日志是不可变且带有时间戳的。
3.  **权限检查**：Lár 的 `SecurityNode` 模式允许您直接在图逻辑中实现权限控制（例如“只有用户 A 能批准工具 B”）。

### 加密审计轨迹 (HMAC 签名)

为了真实符合企业监管要求（如 HIPAA、SOC2、FDA GxP、SEC/FINRA），仅有审计日志是不够的——您必须从数学上证明该日志未被篡改。

Lár v1.5.1 为审计日志引入了**加密签名**。通过向 `GraphExecutor` 传递 `hmac_secret`（例如来自 AWS KMS 或 HashiCorp Vault），引擎将使用 HMAC-SHA256 对最终的 JSON 执行轨迹进行签名。

```python
from lar import GraphExecutor

# 初始化执行器时传入 HMAC 密钥即可开启加密审计
executor = GraphExecutor(
    log_dir="secure_logs", 
    hmac_secret="your_enterprise_secret_key"
)
```

**如何验证（面向审计师）：**
如果载荷中的任何字符（如节点输出、推理字符串或 Token 成本）在执行后被手动修改，签名验证将立即失败。

我们为合规官提供了一个独立的验证脚本，用以从数学上证明日志的真实性：

**步骤 1**：定位生成的 JSON 审计日志（例如 `secure_logs/run_xyz.json`）。
**步骤 2**：获取智能体执行期间使用的企业 HMAC 密钥。
**步骤 3**：从终端运行验证脚本：
```bash
python examples/compliance/11_verify_audit_log.py secure_logs/run_xyz.json your_enterprise_secret_key
```

**结果**：脚本将输出 `[+] VERIFICATION SUCCESSFUL`（真实有效）或 `[-] VERIFICATION FAILED`（已被篡改）。

Lár 包含四个参考实现，展示了该功能在不同行业中的应用：
*   [8_hmac_audit_log.py](../examples/compliance/8_hmac_audit_log.py)（基础用法）
*   [9_high_risk_trading_hmac.py](../examples/compliance/9_high_risk_trading_hmac.py)（算法交易与 FINRA）
*   [10_pharma_clinical_trials_hmac.py](../examples/compliance/10_pharma_clinical_trials_hmac.py)（临床数据路由与 FDA 21 CFR 11）
*   [11_verify_audit_log.py](../examples/compliance/11_verify_audit_log.py)（独立审计员脚本）

## 风险缓解：动态图（自修改代码）

Lár v1.1 引入了 `DynamicNode`，允许智能体在运行时重写其执行拓扑。虽然功能强大，但“自修改代码”在传统意义上是合规性的红灯区。

**Lár 如何缓解这一风险：**

### 1. “代码即事件”原则
在 Lár 中，拓扑结构的变化不是隐藏的内部状态，而是一个显式的**事件 (Event)**。
- `DynamicNode` 输出一个 JSON 格式的 `GraphSpec`。
- 该 JSON 规范在执行前会**物理记录**在审计轨迹中。
- **审计员核实**：审计员可以回溯到智能体决定“添加研究步骤”的确切时刻，并验证*为什么*（基于上下文）。

### 2. 确定性拓扑验证
`TopologyValidator` 是一个**非 AI 的确定性护栏**。
- **白名单**：它强制要求动态生成的节点只能使用预先批准的工具列表。如果 Python 白名单中没有该函数，智能体就*不能*发明一个“删除数据库”工具。
- **循环预防**：它通过数学证明确保新子图是一个 DAG（有向无环图）或有界循环，防止出现“失控智能体”的情况。

### 3. 结构性约束
修改是局部的。`DynamicNode` 只能交换*自身*或其直接下游路径。它不能重写历史或修改上游节点，从而确保“单向”完整性。
