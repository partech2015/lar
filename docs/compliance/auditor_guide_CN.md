# 审计员指南：检查 Lár 智能体

> **适用对象**：合规审查官、质量保证 (QA) 人员以及外部审计师。
> **范围**：本指南阐述了如何根据 **《欧盟 AI 法案》第 13 条（透明度）**，对使用 Lár 引擎构建的“高风险 AI 系统”的行为进行验证。

---

## 1. “玻盒”理念

大多数 AI 系统都是“黑盒” —— 输入一段提示词，输出一个答案，而中间步骤完全不可见。

**Lár 则不同。** 它是一个“玻盒”。
每一个 Lár 智能体都是一个由明确步骤组成的**图 (Graph)**（即流程图）。没有任何事情是“凭空发生”的。

*   **节点 (Nodes)**：流程图中的方框（步骤）。
*   **边 (Edges)**：连接它们的箭头（逻辑）。
*   **状态 (State)**：在节点间传递的内存数据。

作为审计员，您有权检查以上所有三项内容。

---

## 2. 交付物

对于 Lár 智能体的每一次执行，系统都会产出两个交付物。在调查任何事件时，您都应向工程团队索取这些文件。

### A. 图定义 (`architecture.mermaid`)

这是关于*可能发生什么*的地图。它通常是一个 Mermaid 图表或定义节点的 Python 文件。

**检查重点：**

*   **循环**：是否存在无限循环？（Lár 会自动将循环限制在 100 步以内以防止这种情况）。
*   **门控 (Gates)**：在执行敏感操作（如“退款工具”）之前，是否设置了“人工参与回环 (Human-in-the-Loop)”？
*   **分离**： “推理”（LLM）是否与“行动”（工具）分离？

### B. 飞行记录仪 (`state.json` / `flight_recorder.json`)

这是关于*实际发生了什么*的地图。

Lár 会生成一份**状态差异分类账 (State-Diff Ledger)**。它不只是简单地转储日志，而是记录了每一步中确切发生的变化。

**示例追踪轨迹：**

```json
[
  {
    "step": 1,
    "node": "TriageNode",
    "diff": { "classification": "high_risk" }
  },
  {
    "step": 2,
    "node": "SupervisorNode",
    "diff": { "next_action": "escalate_to_human" }
  }
]
```

**验证步骤：**

1.  **可追溯性**：您能否顺着 `step` 编号 1, 2, 3... 进行追踪？
2.  **因果关系**：第 2 步的发生是否是因为第 1 步输出了 "high_risk"（高风险）？
3.  **完整性**：序列中是否存在任何中断或缺失？

---

## 3. 证明确定性

Lár 的“黄金法则”是**确定性**。
*   **规则**：如果您提供相同的输入和相同的随机种子（针对 LLM），智能体**必须**产生完全相同的输出。

**测试方法：**

1.  向工程师索取该事件发生时的**输入状态 (Input State)**。
2.  要求他们使用该输入重新运行智能体。
3.  将新的 `flight_recorder.json` 与原始文件进行对比。
4.  **它们必须完全一致。**

如果不一致，则说明该系统不符合“可重复性”标准。Lár 的设计完全能够通过此项测试。

---

## 4. 常见失败模式

作为审计员，请留意以下常见的“反模式 (Anti-Patterns)”：

*   **“产生幻觉的陪审团”**：在陪审层（提议者 -> 陪审员）中，陪审员是否忽略了违规政策？请检查飞行记录仪中的 `JuryNode` 日志。
*   **“神奇循环”**：智能体是否在循环中空转（规划器 -> 执行器 -> 规划器）而没有取得任何进展？检查 `step` 计数是否达到了上限（100）。
*   **“静默失败”**：某个工具是否失败（例如 API 错误），但 LLM 却忽略了它？检查 JSON 文件中的 `ToolNode` 输出。
